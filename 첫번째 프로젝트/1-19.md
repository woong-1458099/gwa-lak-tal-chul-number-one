## . 반드시 브랜치 생성 후 기능마다 개발

### **cd tikit-backend OR cd tikit-frontend**

### develop 브랜치로 이동하고 task 브랜치를 새로 만들어주셔야 합니다!

### git checkout develop
git pull origin develop

- **브랜치 명명 규칙** (**소문자**만 사용)
    - **형식:** `파트/타입-도메인-상세내용`
    - **파트:** `fe` (Frontend), `be` (Backend), `infra` (Infrastructure)
- **도메인(Domain) 기준**
    - **백엔드:** API의 첫 번째 경로명 (`auth`, `user`, `po` 등)
    
    | **분류** | **도메인명(Domain)** | **예시 브랜치명** | **대상 작업** |
    | --- | --- | --- | --- |
    | **인증/인가** | `auth` | `be/feat-auth-login` | 로그인, 회원가입, 토큰 처리 |
    | **회원 관리** | `user` | `be/fix-user-profile` | 마이페이지, 회원 정보 수정 |
    | **게시판/글** | `post` | `be/feat-post-list` | 게시글 CRUD 로직 |
    | **알림** | `alarm` | `be/feat-alarm-push` | 푸시 알림, 실시간 알림 |
    | **공통/설정** | `common` | `be/chore-common-config` | DB 설정, 전역 예외 처리 |
- **프론트엔드:**
    - 공통 UI: `layout`, `common`, `design`
    - 페이지별: `main`, `search`, `mypage` 등
    
    | **분류** | **도메인명(Domain)** | **예시 브랜치명** | **대상 작업** |
    | --- | --- | --- | --- |
    | **공통 레이아웃** | `layout` | `fe/feat-layout-navbar` | 하단 탭바, 상단 헤더, 사이드바 |
    | **공통 컴포넌트** | `common` | `fe/feat-common-button` | 공통 버튼, 모달, 입력창 |
    | **메인 화면** | `main` | `fe/feat-main-banner` | 홈 화면 UI 및 배너 |
    | **회원/마이페이지** | `user` | `fe/style-user-mypage` | 프로필 화면, 설정 화면 |
    | **설정/통신** | `core` | `fe/chore-core-axios` | API 인스턴스 설정, 라우터 설정 |
- **작업 종류:**
    
    
    | 커밋 유형 | 의미 |
    | --- | --- |
    | `Feat` | 새로운 기능 추가 |
    | `Fix` | 버그 수정 |
    | `Docs` | 문서 수정 |
    | `Style` | 코드 formatting, 세미콜론 누락, 코드 자체의 변경이 없는 경우 |
    | `Refactor` | 코드 리팩토링 |
    | `Test` | 테스트 코드, 리팩토링 테스트 코드 추가 |
    | `Chore` | 패키지 매니저 수정, 그 외 기타 수정 ex) .gitignore |
    | `Design` | CSS 등 사용자 UI 디자인 변경 |
    | `Comment` | 필요한 주석 추가 및 변경 |
    | `Rename` | 파일 또는 폴더 명을 수정하거나 옮기는 작업만인 경우 |
    | `Remove` | 파일을 삭제하는 작업만 수행한 경우 |
    | `!BREAKING CHANGE` | 커다란 API 변경의 경우 |
    | `!HOTFIX` | 급하게 치명적인 버그를 고쳐야 하는 경우 |
- **예시:** `be/feat-auth-login`, `fe/feat-layout-navbar`
- 형식: git checkout -b [브랜치명]
**Ex. git checkout -b be/feat-auth-login**

## 4. 커밋 메시지 및 푸쉬 규칙

- **형식:** `[파트] 타입: 상세내용` (타입은 대문자 시작)
- **이유:** 모노레포 특성상 파트 구분을 최우선으로 하여 가독성을 높임.

### [파트별 커밋 예시]

| **파트** | **예시 커밋 메시지** |
| --- | --- |
| **Backend** | `[BE] Feat: 로그인 API 구현` |
| **Frontend** | `[FE] Style: 메인 페이지 하단바 디자인 수정` |

```bash
# 1. 내 파트 폴더로 이동
cd tikit-backend  # 또는 cd tikit-frontend

# 2. 작업 브랜치 생성 (반드시 develop에서 생성)
git checkout -b be/feat-auth-login

# 3. 코드 수정 후 빌드 테스트 (백엔드 필수)
./gradlew clean build  # BUILD SUCCESS 확인 필수

# 4. 커밋 및 푸시
git add .
git commit -m "[BE] Feat: 로그인 API 구현"
git push origin be/feat-auth-login
```

## 5. 작업 종료 및 로컬 동기화

**머지(Merge)**가 완료된 후, **내 컴퓨터(로컬)**를 최신 상태로 유지하고 다음 작업을 준비하는 필수 과정입니다.

- 로컬 브랜치 삭제 및 마스터 이동

작업이 끝난 브랜치는 로컬에서도 지워주는 것이 깔끔합니다.

```bash
# 1. 디벨롭 브랜치로 복귀
git checkout develop

# 2. 작업이 끝난 로컬 브랜치 삭제 (안전하게 -d 사용)
# 예: git branch -d be/feat-auth-login
git branch -d [방금_작업한_브랜치명]
```

- 디벨롭 브랜치 최신화 (중요!)

다른 팀원이 머지한 최신 코드들을 내 컴퓨터로 가져와서 동기화합니다.

```bash
# 서버의 최신 코드를 내 마스터 브랜치에 반영
git pull origin develop
```

## 6. 새로운 작업 시작

다시 새로운 기능을 만들 때는 반드시 **최신화된 develop**에서 시작해야 합니다.

# 1. (현재 develop 브랜치인지 확인 후) 새로운 브랜치 생성
# 형식: git checkout -b 파트/타입-도메인-상세내용
git checkout -b be/feat-user-logout

# 2. 다시 각자 파트 폴더로 이동하여 개발 시작
cd tikit-backend  # 또는 cd tikit-frontend
