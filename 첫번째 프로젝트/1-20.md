## 지라 사용법

1단계: 출근/작업 시작 (To Do ➔ In Progress)"저 이제 이 일 시작합니다! 건드리지 마세요!"라고 선언하는 단계입니다.티켓 가져오기: To Do(할 일)이나 Backlog에 있는 티켓 중 오늘 할 일을 고릅니다.담당자 할당(Assignee): 담당자를 **'나(Me)'**로 설정합니다. (누가 하는지 모르면 안 되니까요!)상태 변경: 티켓을 드래그해서 **In Progress(진행 중)**으로 옮깁니다.이걸 안 옮기고 일하면, 다른 팀원이 "어? 이거 아무도 안 하네? 내가 해야지" 하고 중복 작업을 하게 됩니다.

2단계: 작업 중 (In Progress)실제 코딩을 하는 단계입니다.스토리 포인트(Story Point) 입력: 만약 팀에서 일정 산정을 한다면, 이 티켓이 얼마나 걸릴지 점수(1점, 3점, 5점 등)를 입력합니다.이슈 발생 시: 작업하다가 기획서가 이상하거나 디자인 파일이 없으면, **댓글(Comment)**로 팀원(@멘션)을 호출해서 물어봅니다. 지라에 기록을 남겨야 나중에 딴소리가 안 나옵니다.

3단계: 작업 완료 & 검사 요청 (In Progress ➔ Review)코딩을 다 했고, 팀원들에게 검사(Merge)를 부탁하는 단계입니다.PR(MR) 생성: 깃랩/깃허브에 코드를 올리고 Merge Request 주소를 복사합니다.상태 변경: 티켓을 Code Review(코드 리뷰) 또는 Review 칸으로 옮깁니다.★댓글 남기기 (가장 중요):Git 자동 연동이 안 되어 있다면, PR 주소 링크를 댓글로 남겨주세요.예시: 작업 완료했습니다.

4단계: 진짜 끝 (Review ➔ Done)코드가 합쳐졌고(Merged), 기능이 잘 돌아가는지 확인까지 끝난 단계입니다.상태 변경: 티켓을 **Done(완료)**으로 옮깁니다.마음의 평화: Done 칸에 쌓인 티켓들을 보며 뿌듯함을 느낍니다. 


지라 사용 예시 (모범 답안)상황: 작성자님이 기본 틀 구성 티켓을 맡아서 처리할 때순서행동지라 상태(Status)1"나 이거 할게!" 하고 내 얼굴 박고 오른쪽으로 옮김In Progress2열심히 코드 짜고 Git에 Push 함In Progress3PR 올리고, 지라 티켓에 "PR 링크" 댓글로 남김In Progress4상태를 리뷰 단계로 옮기고 팀원한테 "리뷰해줘" 말함In Progress ➔ Review5(팀원이 Merge 해줌)Review6최종 확인 후 완료 처리Review ➔ Done


### 1. Issue 생성 (GilLab 웹)
📝 1. Title (제목)
가장 중요합니다. [분야] 작업 내용 형식으로 적는 게 국룰입니다.

입력할 내용: [FE] 출입증 화면 생체 인증 로직 연결

(팁: [FE]는 프론트엔드라는 뜻입니다. 나중에 백엔드 이슈랑 섞여도 찾기 쉽습니다.)

📄 2. Description (설명)
여기가 핵심입니다. **"무엇을 하면 이 작업이 끝나는가?"**를 체크리스트로 적어주세요. 마크다운 문법을 쓰면 체크박스가 생겨서 아주 멋집니다. 아래 내용을 그대로 복사해서 붙여넣어 보세요.

입력할 내용:

Markdown

## 🎯 작업 목적
- 라이브러리(react-native-biometrics)를 사용하여 출입증 화면에서 실제 생체 인증을 수행한다.

## ✅ 할 일 (Checklist)
- [ ] `react-native-biometrics` 라이브러리 설치 및 설정
- [ ] `useBiometric` 훅을 `AccessCardScreen`에 연결
- [ ] 인증 성공 시 화면 UI 변경 (빨강 -> 초록) 확인
- [ ] Android/iOS 에뮬레이터 테스트 완료
👤 3. Assignee (담당자)
**"이거 누가 하나요?"**라는 질문에 답하는 곳입니다.

행동: 입력창을 클릭하고 **Assign to me**를 누르거나, 본인 아이디를 검색해서 선택하세요.

(팁: 이걸 비워두면 "주인 없는 고아 이슈"가 됩니다. 꼭 본인을 찍으세요!)

🏷️ 4. Labels (라벨) - 선택사항
이슈에 색깔 태그를 붙여서 분류하는 겁니다.

행동: Frontend, Feature(기능) 같은 라벨이 있다면 선택해주세요. 없으면 넘어가도 됩니다.

🚀 5. 마지막: [Create issue] 버튼 클릭!
다 채우셨으면 하단의 파란색 [Create issue] 버튼을 쾅! 누르세요.

🎉 6. 결과 확인 (가장 중요!)
화면이 바뀌면서 방금 만든 이슈가 보일 겁니다. 여기서 제목 옆에 있는 작은 숫자를 찾으세요.

예: [FE] 출입증 화면... #12

여기서 #12 (숫자는 다를 수 있음)가 바로 작성자님의 작업 번호입니다. 이제 터미널로 가서 브랜치를 만들 때 이 번호를 쓰시면 됩니다.

Bash

git checkout -b fe/feat/12-biometric-connect
(여기서 12 자리에 방금 본 그 숫자를 넣는 겁니다!)



2. Branch 생성 (터미널)
이슈 번호(10)를 따서 브랜치를 만듭니다. 이렇게 하면 나중에 찾기 쉽습니다.

명령어:

Bash

# git checkout -b [타입]/[이슈번호]-[설명]
git checkout -b fe/feat/10-biometric-connect
3. 작업 및 저장 (코딩)
열심히 코드를 짭니다. (AccessCardScreen.tsx 수정 등)

4. Commit (내 컴퓨터에 저장)
★핵심: 커밋 메시지 맨 앞에 **#이슈번호**를 붙입니다.

명령어:

Bash

git add .
git commit -m "feat: #10 생체 인증 훅 화면 연결 및 UI 적용"
(이렇게 하면 GitLab 이슈 페이지에 "방금 이 커밋이 올라왔어요!" 하고 자동으로 링크가 달립니다.)

5. Push (브랜치에 저장/서버 전송)
작업한 내용을 GitLab 서버의 해당 브랜치로 보냅니다.

명령어:

Bash

git push origin fe/feat/10-biometric-connect
🏁 그 다음은요? (마무리)
여기까지가 질문하신 "브랜치에 저장" 단계입니다. 아주 잘하셨습니다! 👍 하지만 일이 완전히 끝나려면 마지막 한 단계가 남습니다.

Merge Request (MR): GitLab 웹페이지에 가면 "방금 푸시한 브랜치로 MR 만들래?"라고 버튼이 뜹니다. 그거 눌러서 "제 코드(fe/feat/...)를 원본(develop)에 합쳐주세요!" 라고 요청하면 진짜 끝입니다.
